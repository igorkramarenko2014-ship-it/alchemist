## Problem
Current output has a single survivalRates object (isValid / distribution / adversarial).
This hides WHICH gate is the bottleneck and WHICH provider is causing failures.
We need per-gate, per-model breakdown to turn calibration into diagnostics.

## What to add to calibrate-gates.ts

### 1. Per-gate rejection counts (add alongside existing survivalRates)
For each raw candidate, track which specific gate rejects it first:
- "reasoning_length"   — reasoning.trim().length < 20
- "score_range"        — score not in [0,1]
- "state_missing"      — state falsy
- "distribution"       — candidatePassesDistributionGate fails
- "adversarial"        — candidatePassesAdversarial fails
- "passed_all"         — survived all gates

Shape to add to output JSON:
"gateBreakdown": {
  "reasoning_length":  { "count": 0, "rate": 0 },
  "score_range":       { "count": 0, "rate": 0 },
  "state_missing":     { "count": 0, "rate": 0 },
  "distribution":      { "count": 0, "rate": 0 },
  "adversarial":       { "count": 0, "rate": 0 },
  "passed_all":        { "count": 0, "rate": 0 }
}
(rate = count / totalRawCandidates, 0–1)

### 2. Per-model survival rates
For each panelist (DEEPSEEK / LLAMA / QWEN), track:
- rawCount: how many candidates it contributed
- passedAll: how many survived all gates
- survivalRate: passedAll / rawCount

Shape to add:
"perModelStats": {
  "DEEPSEEK": { "rawCount": 0, "passedAll": 0, "survivalRate": 0 },
  "LLAMA":    { "rawCount": 0, "passedAll": 0, "survivalRate": 0 },
  "QWEN":     { "rawCount": 0, "passedAll": 0, "survivalRate": 0 }
}

### 3. Cross-gate interaction note (comment in output, not computed)

Tablebase as scored competitor
Currently the keyword tablebase hard-bypasses runTriad entirely. This makes it a scored entrant instead — tablebase candidates go through scoreCandidates alongside triad output and compete on weightedScore. Best candidates win regardless of source.
Why this matters: A tablebase row that short-circuits the triad means a stale, pre-validated candidate can silently win over a better live LLM candidate — with no competition. Making it a scored entrant preserves its value as prior knowledge while letting the triad beat it when the live output is better. Telemetry already distinguishes the source via mode: "tablebase" vs "fetcher".

## Current behavior (triad.ts)
When lookupTablebaseCandidate matches a prompt, runTriad short-circuits:
returns tablebase candidates directly without calling panelist fetchers.
Telemetry: preset_tablebase_hit + mode: "tablebase".

## Problem
A tablebase hit silently wins over any live LLM candidate — no competition.
If the tablebase row is stale or low-quality, there is no recovery path.

## New behavior (opt-in flag)
Add optional flag to RunTriadOptions:
  tablebaseMode?: 'shortcircuit' | 'compete'
  Default: 'shortcircuit'  ← preserves existing behavior when flag absent

When tablebaseMode === 'compete':
1. Run tablebase lookup as normal — collect matching rows as AICandidate[]
   Tag each with panelist: 'TABLEBASE' (or use existing tablebase panelist field
   if already typed — check reliability/tablebase-schema.ts)
2. ALSO run the full panelist fetcher path (all three providers, same timeout)
3. Merge tablebase candidates + panelist candidates into one pool
4. Run the full gate pipeline: filterValid → distribution → adversarial → slice(MAX_CANDIDATES)
5. scoreCandidates handles Slavic dedupe + weightedScore ranking downstream (unchanged)
6. Telemetry:
   - preset_tablebase_hit still fires when tablebase has matches
   - triad_run_end mode: "compete" (new mode value alongside stub/fetcher/tablebase)
   - Add tablebaseCandidateCount + triadCandidateCount to triad_run_end payload
