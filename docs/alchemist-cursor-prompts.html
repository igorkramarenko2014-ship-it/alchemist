## What to look at in tools/gate-calibration-output.json

survivalRates.isValid
  > 0.7 = good  |  0.3–0.7 = investigate  |  < 0.3 = STOP, reasoning prompt needs fix

survivalRates.distribution (Undercover gate)
  > 0.7 = good  |  < 0.5 = paramArray distribution too extreme, tune system prompt

survivalRates.adversarial
  > 0.7 = good  |  < 0.5 = entropy/variance too low, tune system prompt

slavicDedupe.uniqueAfterDedupe / validCandidatesIn
  0.3–0.5 = healthy (expected overlap between panelists)
  < 0.2 = panelists producing near-identical output

providerFailureRate per panelist
  0 = good  |  > 0.2 = timeout or parse failures, check latency

reasoningLengths.pctUnder20
  0 = good  |  > 0 = some candidates silently dropped

runDurationMs.mean
  < 3000 = fast  |  3000–7000 = normal  |  > 7000 = close to timeout wall

## If survivalRates are all > 0.7
You have a working triad. Proceed to P3 threshold review (optional).

## If any survivalRate < 0.3
Do NOT change gate constants yet. First:
1. Look at raw calibration output for patterns
2. Adjust the system prompt in the affected route
3. Re-run calibration
4. Only change constants.ts thresholds if prompt tuning fails,
   and only after updating both FIRE.md + FIRESTARTER.md + tests
requires: all 3 keys in .env.local + pnpm dev running no code changes — terminal only check tools/gate-calibration-output.json after run
